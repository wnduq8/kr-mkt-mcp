## 한 줄 요약

Google Cloud가 엔터프라이즈 멀티 에이전트 시스템의 두 축을 **A2A(Agent-to-Agent, 에이전트 간 통신)** 와 **MCP(Model Context Protocol, 에이전트–도구·데이터 접근)** 로 분리하고, 둘을 결합한 **5가지 통합 패턴**을 제시했다.

## 핵심 개념: A2A vs MCP

| 구분 | 역할 | 다루는 대상 |
|------|------|------------|
| **A2A** (Agent-to-Agent) | 에이전트 ↔ 에이전트 통신 프로토콜 | 다른 에이전트 |
| **MCP** (Model Context Protocol) | 에이전트 ↔ 도구·데이터 접근 프로토콜 | API, DB, 파일, SaaS 등 비-에이전트 리소스 |

핵심 설계 원칙: **에이전트 간 통신(A2A)과 도구 접근(MCP)을 계층적으로 분리**한다. 한 프로토콜이 모든 걸 떠안지 않는다.

## 5가지 통합 패턴

### 패턴 1. 에이전트 발견과 등록 (Discovery & Registry)

- **Agent Card**: 에이전트가 자신의 기능·인증 방식·rate limit·제약을 JSON으로 공개하는 일종의 "명함"
- **Agent Registry**: 중앙 레지스트리에서 기능 기반 검색·접근 통제
- 시사점: 에이전트가 **사람이 카탈로그를 보고 고르는 방식이 아니라**, 다른 에이전트가 기능 메타데이터로 찾아 쓸 수 있어야 한다

### 패턴 2. 크로스팀 위임 (Cross-team Delegation)

- 서로 다른 언어(Go, Java, TypeScript 등)로 작성된 에이전트들이 A2A로 협업
- 각 팀이 **독립 배포·독립 진화** 가능
- 시사점: 에이전트는 마이크로서비스의 다음 단계 — 언어/스택 자유, 인터페이스(A2A)만 지키면 됨

### 패턴 3. MCP를 통한 도구 연결 (Tools via MCP)

- 단일 프로토콜(MCP)로 이질적 데이터 소스/도구 통합
- **기존 API 거버넌스(Apigee 등) 재활용** — 인증·로깅·rate limit을 새로 짜지 않고 그대로 얹음
- Google Cloud는 **60개 이상 사전 구축 도구** 제공
- 시사점: MCP를 채택하면 도구 추가가 "API 한 번 거버넌스 통과 → 모든 에이전트가 자동 사용"으로 단순화

### 패턴 4. 조직 간 협업 (Inter-org Collaboration)

- **Agent Gallery**: 100개 이상의 검증된 파트너 에이전트 마켓플레이스
- **Agent Gateway**: 외부 에이전트 호출에 세밀한 접근 제어(권한·할당량·감사) 적용
- 시사점: B2B에서 다른 조직의 에이전트를 호출하는 시대가 시작 — 게이트웨이가 곧 트러스트 경계

### 패턴 5. 이벤트 기반 에이전트 메시 (Event-driven Mesh)

- **BigQuery, Pub/Sub** 기반 이벤트 감지 → 적합한 에이전트로 자동 위임
- "메시(mesh)"는 **자기 조직화** 구조 — 사람이 워크플로우를 일일이 그리지 않음
- 시사점: 단발성 chat-call 형 에이전트에서 → **상시 가동되며 이벤트에 반응**하는 형태로 진화

## 차별점 (Google이 강조한 포인트)

- 특정 프레임워크/클라우드에 묶이지 않는 **개방형 설계** 지향 (단, 실제 구현은 GCP 생태계에 의존적)
- A2A와 MCP의 **계층 분리** — 통신과 도구 접근을 한 프로토콜에 욱여넣지 않음
- **기존 인프라 재사용** — Apigee(API 거버넌스), BigQuery(데이터), Pub/Sub(이벤트)을 그대로 활용

## 주의사항 / 비판적 관점

- "개방형"이라 해도 실제 통합은 **GCP 인프라 의존**. 멀티클라우드 환경에서의 개방성은 검증 필요
- 분산 에이전트 시스템 = **분산 시스템의 모든 어려움 그대로** 상속 (의존성 관리, 장애 전파, 디버깅 난이도)
- A2A·MCP가 표준이 되려면 Google 외 다른 벤더의 채택이 관건

## 한국 마케팅 데이터 MCP 프로젝트(`kr_mkt_mcp`) 관점에서의 시사점

이 프로젝트가 결국 만들려는 것이 "광고 데이터를 AI 도구에 넘기는 MCP"이므로, 위 다섯 패턴 중 직접 적용 가능한 것:

1. **패턴 3 (MCP 도구 연결)** — 본 프로젝트의 본질 그 자체. 광고 플랫폼별 MCP를 도구로 노출
2. **패턴 1 (Agent Card 발견)** — MCP 서버 자체가 자기 capability를 JSON으로 명세 → AI 클라이언트가 자동 발견
3. **패턴 5 (이벤트 메시)** — 광고 성과 임계치(예: ROAS 급락) 이벤트 → 분석 에이전트 자동 호출하는 응용 방향
4. 본 프로젝트에서 당장은 **A2A 불필요** — 1단계는 단일 MCP 서버 + Claude Desktop/Codex 클라이언트 구조면 충분
